发布于：Feb 22, 2025

Spring是一款：Java EE 应用框架。

它尽可能多的实现了构建应用程序所需要的底层代码，使得开发者只需专注于系统功能和业务逻辑的开发。

学习Spring之前，先了解什么是容器。

Spring的核心就是提供了一个IoC容器，它可以管理所有轻量级的JavaBean组件，提供的底层服务包括组件的生命周期管理、配置和组装服务、AOP支持，以及建立在AOP基础上的声明式事务服务等。

JavaBean是什么：

JavaBean 是一种可重用组件。为写成JavaBean，类必须是具体的和公共的，并且具有无参数的构造器。JavaBean 通过提供符合一致性设计模式的公共方法将内部域暴露成员属性，set和get方法获取。

这就是个简单的JavaBean了，其实还有一些其他的定义，但我们不用考虑的那么麻烦。

什么是IOC？

IoC 是 Inversion of Control 的简写，译为“控制反转”，它不是一门技术，而是一种设计思想，是一个重要的面向对象编程法则，能够指导我们如何设计出低耦合、更优良的程序。

  • 控制反转，反转的是什么？
    • 将对象的创建权利交出去，交给第三方容器负责。
    • 将对象和对象之间关系的维护权交出去，交给第三方容器负责。
  • 控制反转是一种思想。
  • 控制反转是为了降低程序耦合度，提高程序扩展力。
  • 控制反转这种思想如何实现呢？
    • DI（Dependency Injection）：依赖注入

Spring 通过 IoC 容器来管理所有 Java 对象的实例化和初始化，控制对象与对象之间的依赖关系。我们将由 IoC 容器管理的 Java 对象称为 Spring Bean，它与使用关键字 new 创建的 Java 对象没有任何区别。

IoC 容器是 Spring 框架中最重要的核心组件之一，它贯穿了 Spring 从诞生到成长的整个过程。

ApplicationContext是什么?

ApplicationContext接口

BeanFactory 和 ApplicationContext接口，代表 Spring IoC 容器。其中，BeanFactory 是访问 Spring 容器的根接口。它提供了管理 Bean 的基本功能。

而 ApplicationContext 是 BeanFactory 的子接口。因此，它具备 BeanFactory 的所有功能。

BeanFactory和ApplicationContext的区别

BeanFactory的实现是按需创建，即第一次获取Bean时才创建这个Bean

ApplicationContext会一次性创建所有的Bean

定制Bean

Bean 的作用域（Scope）

Bean 的作用域（Scope）**定义了 Bean 的生命周期和它在容器中的创建方式。其中,Singleton 和 Prototype 是 Spring 中常用的两种作用域，它们之间有以下区别：

Singleton 作用域
  • 定义：Singleton 是 Spring 中的默认作用域。对于 Singleton 作用域的 Bean，Spring 容器在启动时会创建该 Bean 的唯一实例，并在整个应用程序的生命周期内共享该实例。
  • 创建时机：容器启动时就会创建这个 Bean 实例（急切初始化）。
  • 生命周期：整个 Spring 容器的生命周期内，Singleton Bean 只有一个实例。
  • 线程安全：由于 Singleton Bean 是共享的，必须保证它的线程安全。通常，Singleton Bean 需要是无状态的，或者通过适当的同步机制来保证线程安全。

使用场景：

  • 共享资源：适合需要共享的资源，比如数据库连接池、日志管理器等。
  • 无状态服务：如果 Bean 没有维护任何状态（或状态在其他地方），Singleton 是理想选择。比如常用的工具类、配置类等。
Prototype 作用域
  • 定义：Prototype 作用域表示 Spring 容器会在每次请求时创建该 Bean 的一个新实例。每次从容器中获取该 Bean 时，都会返回一个新的实例。
  • 创建时机：只有在每次调用 getBean() 方法时才会创建 Bean 实例（懒加载）。
  • 生命周期：每次获取 Bean 时都会创建一个新的实例，Spring 容器并不负责管理 Prototype Bean 的销毁。
  • 线程安全：由于每次获取 Bean 都是新的实例，因此 Prototype Bean 本身无需考虑线程安全的问题。

示例：

使用场景：

  • 有状态服务：如果每次请求需要一个独立的 Bean 实例，并且该 Bean 有自己的状态，可以使用 Prototype。例如，用户会话管理、临时计算服务等。
  • 需要独立生命周期的 Bean：如果 Bean 的生命周期与请求的生命周期绑定（每次请求时创建新的实例），就可以考虑 Prototype。
使用场景总结
  • Singleton 适合的场景：
    • 共享资源：例如，数据库连接池、线程池、缓存管理、日志工具等，都是单例的，避免重复创建和资源浪费。
    • 无状态 Bean：如果一个 Bean 没有自己的状态，且所有客户端共享这一个 Bean 实例，那么 Singleton 适合它。
    • 应用配置类：例如读取配置文件的类、工具类等。
  • Prototype 适合的场景：
    • 有状态 Bean：例如，用户的会话对象、临时存储计算结果的对象等，它们需要每次都创建一个新的实例。
    • 请求级别的 Bean：如果 Bean 与请求或线程生命周期挂钩（例如，处理请求的服务对象），那么使用 Prototype 作用域可以确保每次请求时创建一个新实例。

AOP

AOP是Aspect Oriented Programming，即面向切面编程。

通过预编译方式和运行期动态代理的方式，在不修改源代码的情况下，给程序动态统一添加额外功能的一种技术。

AOP可以对业务逻辑的各个部分进行隔离，从而使得业务逻辑各部分之间的耦合性降低，提高程序的可重用性，同时提高开发效率。

Spring实现AOP原理：目标对象和切面都是普通Java类，通过JVM的动态代理功能或者第三方库实现运行期动态织入。

Spring的AOP实现就是基于JVM的动态代理。由于JVM的动态代理要求必须实现接口，如果一个普通类没有业务接口，就需要通过CGLIB或者Javassist这些第三方库实现。

AOP技术看上去比较神秘，但实际上，它本质就是一个动态代理，让我们把一些常用功能如权限检查、日志、事务等，从每个业务方法中剥离出来。

拦截器类型

顾名思义，拦截器有以下类型：

  • @Before：这种拦截器先执行拦截代码，再执行目标代码。如果拦截器抛异常，那么目标代码就不执行了；
  • @After：这种拦截器先执行目标代码，再执行拦截器代码。无论目标代码是否抛异常，拦截器代码都会执行；
  • @AfterReturning：和@After不同的是，只有当目标代码正常返回时，才执行拦截器代码；
  • @AfterThrowing：和@After不同的是，只有当目标代码抛出了异常时，才执行拦截器代码；
  • @Around：能完全控制目标代码是否执行，并可以在执行前后、抛异常后执行任意拦截代码，可以说是包含了上面所有功能。
